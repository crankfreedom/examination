# ExamHub 系统设计原则（宪法）

> **文档版本：** v1.2.0
> **最后更新：** 2026-07-13
> **本文档为 ExamHub 的最高设计准则，所有架构决策和代码实现必须遵守。**

---

## 第一条：Content First

> 围绕 Content（内容）设计整个系统，不围绕 PDF 设计。

- 核心数据模型是 **Question（题目）**，不是 PDF 文件
- PDF 只是渲染结果，不是数据源
- 所有业务流程应基于 Question → Paper → Collection 的内容链路
- 任何新增功能都要问：这是围绕 Content 设计，还是围绕 PDF 设计？

---

## 第二条：唯一编号

> 所有对象统一编号系统，确保全局唯一、可追溯。

| 前缀 | 对象 | 示例 |
|------|------|------|
| Q | Question（题目） | Q000001 |
| P | Paper（试卷） | P000001 |
| C | Collection（合集） | C000001 |
| PR | Product（商品） | PR000001 |
| O | Order（订单） | O000001 |
| U | User（用户） | U000001 |
| M | Material（材料） | M000001 |

- 前缀 + 6 位数字，补齐零
- 编号不可复用（删除的记录编号作废）
- 编号由统一服务生成，不依赖数据库自增

---

## 第三条：内容与商品分离

> Content ≠ Product，内容和销售体系完全独立。

- Content（Question / Paper / Collection）只关注内容本身
- Product（商品）关注定价、促销、销售策略
- 一个 Collection 可以生成多个商品：
  - 普通商品
  - VIP 专属商品
  - 限时促销商品
  - 积分兑换商品
- 商品可以绑定多个 Content 资源（Paper / Collection）

---

## 第四条：统一导入器

> 所有导入格式最终转换为同一 JSON Schema。

- 支持 JSON / Markdown / Word（DOCX）三种格式
- 统一转换流程：上传 → 解析 → **预览** → 发现错误 → 修改 → **确认** → 批量写入
- V1 不使用 AI 识别，全部由管理员后台导入或 API 批量录入
- 每个导入生成解析报告（成功题数 / 失败题数 / 图片数 / 公式数 / 材料组数）

---

### 原则 4.5：自动化采集

> 通过 Chalk 采集器自动化获取外部题库内容，与 Importer 手动导入互为补充。

- Chalk 通过 Selenium WebDriver 自动化爬取第三方题库网站（如粉笔网）
- 采集结果保存为中间数据文件，后续通过 Importer 统一导入
- 第三方网站凭证通过环境变量注入，不硬编码
- 采集任务运行在独立子进程中，不阻塞主服务
- 采集的数据需经管理员审核后方可发布

## 第五条：PDF 只是渲染结果

> PDF 不是数据源，只是内容的动态渲染产物。

- 永远由 Question → Paper → Render → 动态生成
- PDF 不提前生成，按需生成 + 缓存到 OSS
- 题目更新后自动清除缓存，重新生成
- 已购用户可永久免费更新 PDF

---

## 第六条：版本管理

> 所有资源支持版本控制。

- 文档版本：docs 目录整体版本号（v1.0.0）
- 内容版本：Question / Paper / Collection 支持版本迭代
- Product 版本：商品信息变更管理
- PDF 版本：每次内容更新生成新版本
- 下载中心显示版本信息，用户可查看更新历史

---

## 第七条：页面兼容性

> 所有页面后续可平滑升级为在线刷题。

- 题目展示组件同时支持"阅读模式"和"刷题模式"
- 答案和解析的渲染需要考虑选择题、填空题、主观题、判断题等多种题型
- 页面组件设计时预留交互状态（选中、提交、判分、显示解析）
- V1 的"在线阅读页"就是 V2 "在线刷题页"的基础

---

## 第八条：SEO 先行

> URL 一旦确定永不改变。

- 每个资源有独立的、语义化的永久 URL
  - /paper/P000001
  - /collection/C000001
  - /product/PR000001
- 支持自定义 SEO 信息：Title / Description / Keywords
- 搜索引擎友好的 URL 结构
- 自动生成 Sitemap 和 Robots.txt
- URL 变更需要 301 重定向和更新 Sitemap

---

## 补充原则

### 原则 9：安全优先

- 用户密码加密存储（bcrypt）
- PDF 传输加密（AES-GCM）
- DownloadToken 一次性、5 分钟有效期
- 明水印 + 暗水印保护（含用户名、手机号、订单号、下载时间）
- 所有接口需要权限校验

### 原则 10：渐进增强

- V1 只做核心功能，不追求大而全
- 功能按照 V1 → V2 → V3 路线图逐步实现
- 每个版本都是可交付、可运行的完整产品
- 不要在 V1 做 V2 才需要的事，但要为 V2 预留扩展点

### 原则 11：开放与上云

- 对象存储 OSS 作为文件存储底座
- 后续支持 CDN 加速
- 搜索后续接入 Elasticsearch / Meilisearch
- 模块化设计，支持渐进式迁移到微服务（如有需要）

### 原则 12：AI 约束

> 任何 Agent 不得修改 docs/ 中已确认的内容。

- 文档修改需要明确的版本升级
- 文档修改须通过 ADR（架构决策记录）流程
- 修改后必须更新版本号

---

*本文档是 ExamHub 的最高设计准则，任何偏离这些原则的决策都需要通过 RFC 和 ADR 流程，并有充分的理由说明为何需要偏离。*
