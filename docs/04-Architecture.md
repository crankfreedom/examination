## 二、分层架构

### 2.1 表示层（前端）

- **用户端**：13 个页面覆盖浏览、搜索、购买、下载、个人中心等完整流程
- **管理端**：13 个管理模块覆盖内容管理、订单管理、用户管理、系统设置等
- **SEO 渲染**：关键页面支持服务端渲染（SSR）或静态生成（SSG）
- **PDF 客户端渲染**：前端解密后通过 pdf-lib/pdfmake 在浏览器端生成 PDF

### 2.2 业务模块层

按业务领域划分的模块，每个模块包含：

| 组件 | 说明 |
|------|------|
| **Controller** | 处理 HTTP 请求，参数校验，返回响应 |
| **Service** | 业务逻辑实现 |
| **Repository** | 数据访问层（数据库操作） |
| **DTO / Schema** | 数据传输对象和校验规则 |
| **Types / Interfaces** | TypeScript 类型定义 |

### 2.3 公共服务层

跨模块共享的基础服务：

| 服务 | 职责 |
|------|------|
| 编号生成器 | 生成全局唯一编号（Q000001, P000001...） |
| 加密服务 | AES-GCM 加密/解密、JWT 签发验证 |
| 缓存服务 | Redis 缓存封装 |
| 事件总线 | 模块间事件通信（如：下单成功 → 触发下载权限更新） |

### 2.4 基础设施层

| 组件 | 用途 |
|------|------|
| 关系型数据库 | 业务数据持久化 |
| Redis | 缓存、Session、Token、分布式锁 |
| OSS | PDF 缓存、图片、静态资源文件存储 |
| CDN | 加速静态资源和 PDF 下载 |

## 三、核心数据流

### 3.1 用户购买流程

\\\
游客 → 浏览商品详情页 → 点击购买 → 微信扫码支付
  → 微信返回 openid → 自动创建账号 → 绑定订单
  → 自动生成 JWT → 自动登录 → 跳转到下载页 → 下载 PDF
\\\

**关键设计**：用户几乎无感知，实际上后台已经注册了账号。后续通过手机号即可找回所有订单。

### 3.2 PDF 下载流程

\\\
Question → Paper JSON → AES-GCM 加密 → HTTPS 传给前端
  → 浏览器内存解密 → HTML Render → 添加动态水印
  → pdf-lib/pdfmake 生成 PDF → 下载到本地
\\\

### 3.3 内容导入流程

\\\
上传文件（JSON / Markdown / DOCX）
  → 解析器解析 → 转换为统一 JSON Schema
  → 预览（显示题目列表、发现错误）
  → 管理员修改 → 确认导入
  → 生成编号 → 批量写入数据库
  → 生成解析报告
\\\


### 3.4 自动化采集流程

\\\
管理员触发 /chalk/create/examination
  -> Access 层 fork 子进程
  -> Selenium WebDriver 登录目标网站
  -> 智能组卷 -> 提交试卷 -> 获取答案版
  -> 逐题提取（题干/选项/答案/解析/考点/来源/统计）
  -> 保存为本地 JS 文件（中间产物）
  -> 后续通过 Importer 导入内容中心
\\\

### 3.5 搜索流程

\\\
用户输入关键词
  → V1：数据库 LIKE 查询（全文索引）
  → V2+：Elasticsearch / Meilisearch
  → 搜索结果分为两组：试卷/合集/商品 + 资讯/公告/知识点
  → 支持多维筛选：考试类型、年份、地区、科目、资料类型、价格、难度、题量
\\\

## 四、模块间通信

### 4.1 直接调用（同步）

同一进程内，Service 层直接调用其他模块的 Service 方法。

\\\
// 示例：下单时调用用户模块检查 VIP 状态
class OrderService {
  async createOrder(userId: string, productId: string) {
    const user = await UserService.getUserById(userId);
    const vipInfo = await VipService.getVipInfo(userId);
    // ... 业务逻辑
  }
}
\\\

### 4.2 事件总线（异步）

跨模块的异步通知通过事件总线实现。

\\\
// 示例：支付成功后触发多个动作
EventBus.emit('order.paid', { orderId: 'O000001' });

// 监听：
// 1. 下载模块：为该用户生成下载权限
// 2. 用户模块：更新用户积分
// 3. 统计模块：更新销售额统计
// 4. 日志模块：记录支付日志
\\\

## 六、数据架构

### 6.1 内容模型关系

\\\
Material   → 1:N → Question    （一个材料关联多道题）
Question   → N:N → Paper       （多对多，通过试卷题目关联表，支持材料题分组）
Paper      → N:N → Collection  （多对多，通过合集试卷关联表）
Collection → 1:N → Product     （一个合集生成多个商品）
Product    → 1:N → Order       （一个商品产生多个订单）
Order      → 1:1 → User        （一个订单属于一个用户）
\\\

### 6.2 架构约束

| 约束 | 说明 |
|------|------|
| 模块依赖方向 | 公共服务层 ← 业务模块层，业务模块之间可调用 Service 但禁止循环依赖 |
| 数据访问 | 每个模块只能操作自己的 Repository，跨模块数据访问需通过 Service 接口 |
| 事件使用场景 | 仅用于跨模块的异步通知，不用于核心业务同步流程 |
| 配置管理 | 所有配置通过环境变量注入，不硬编码在代码中 |
| 错误处理 | 全局错误处理中间件统一处理，模块内只抛出业务异常 |
