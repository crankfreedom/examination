# RFC-0001：ExamHub RFC 流程规范

> **状态：** 已采纳
> **提出日期：** 2026-07-10
> **最后更新：** 2026-07-13

---

## 摘要

定义 ExamHub RFC（Request for Comments，改进提案）流程，用于提议、讨论和记录项目的重大变更。

---

## 动机

当需要变更已确认的设计文档、引入重大新功能、修改架构决策时，需要一个正式的提案和讨论流程。

---

## RFC 流程

### 阶段 1：提交

1. 在 docs/11-RFC/ 创建 RFC 文档
2. 编号规则：RFC-{3位数字}（从 001 开始）
3. 使用以下模板：

`markdown
# RFC-{编号}：{标题}

> **状态：** 草稿
> **提出日期：** {日期}
> **提出人：** {姓名/Agent}

## 摘要
{用 2-3 句话描述提案内容}

## 动机
{为什么要做这个变更？解决了什么问题？}

## 详细设计
{具体的变更方案，包括技术细节、接口变更、数据模型变更等}

## 备选方案
{列出考虑的其它方案及不选择的原因}

## 影响分析
{对现有系统、文档、用户的影响}

## 兼容性
{向后兼容性分析}

## 实施计划
{分步骤的实施计划}
`

### 阶段 2：讨论

- 提案提交后进入讨论期（至少 3 天）
- 相关人员/Agent 可提出修改意见
- 提案人根据反馈更新提案

### 阶段 3：决策

| 结果 | 说明 |
|------|------|
| 已采纳 | 提案通过，进入实施阶段 |
| 已拒绝 | 提案被拒绝，记录原因 |
| 已搁置 | 当前不实施，后续重新评估 |

### 阶段 4：实施

- 已采纳的 RFC 更新状态为"实施中"
- 实施完成后更新状态为"已完成"
- 记录到 ADR（如涉及架构决策）

---

## RFC 适用范围

| 需要 RFC | 不需要 RFC |
|---------|------------|
| 修改已确认的 docs 内容 | Bug 修复 |
| 新增重大功能 | 小范围优化 |
| 修改架构决策 | 文档勘误（错别字等） |
| 引入新依赖（重大） | 测试用例补充 |
| 修改编号规则 | 代码重构（不改变行为） |
