# ADR-0001：采用模块化单体（Modular Monolith）架构

> **状态：** 已采纳
> **日期：** 2026-07-10

---

## 背景

ExamHub 是一个以内容为核心的考试平台，需要支持内容管理、商品销售、在线刷题等多种功能。初期团队规模有限，但系统需要支持后续向 V2、V3 演进。

---

## 决策

采用**模块化单体（Modular Monolith）**架构。所有功能部署在同一个进程中，但按业务逻辑划分为清晰的模块，模块之间有明确的接口边界。

---

## 动机

| 因素 | 说明 |
|------|------|
| 团队规模 | 初期团队 1-3 人，微服务运维成本高 |
| 部署复杂度 | 单体应用只需部署一个实例 |
| 开发效率 | 模块间调用效率高，无需网络通信 |
| 数据一致性 | 单一数据库，无分布式事务 |
| 演进能力 | 模块化设计便于后续按需拆分微服务 |
| 技术成熟度 | Express + TypeScript + 模块化已足够 |

---

## 备选方案

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|:----:|
| 模块化单体 | 低复杂度、高开发效率、单库事务 | 扩展受限于单机（可水平扩容） | ✓ |
| 微服务 | 独立扩展、独立部署 | 分布式复杂度、网络延迟、数据一致性 | V2 后评估 |
| Serverless | 免运维 | 冷启动、运行成本不确定 | 不适用 |

---

## 影响

### 正面

- 开发效率高，快速迭代
- 部署简单，监控容易
- 数据一致性保障

### 负面

- 单点风险（可通过负载均衡缓解）
- 技术栈绑定（Node.js）
- 扩展受限于单机资源

---

## 实施

1. 在 ackend/ 下按模块组织目录
2. 每个模块包含 Controller / Service / Repository
3. common/ 下放公共服务（ID 生成、加密、缓存）
4. 模块间通过 Service 接口调用，禁止直接访问数据库
5. 使用事件总线处理跨模块异步通知
